# iOS — App Lifecycle, Notifications, Security, Deep Linking

### AppDelegate/SceneDelegate methods and application states
**They ask:** "Walk through the app's states and which delegate methods fire at each transition."

iOS models an app's presence on screen and in memory as a small state machine — **Not Running → Inactive → Active → Background → Suspended** — because the OS needs well-defined hooks to save work before it can reclaim memory or CPU from an app the user isn't looking at. `AppDelegate` owns process-level lifecycle (`didFinishLaunchingWithOptions` — one-time setup, still fires even in a multi-scene app); `SceneDelegate` owns *per-window* UI lifecycle since multi-window iPad support split "the app launched" from "a scene became visible."

```swift
func sceneDidBecomeActive(_ scene: UIScene) { }       // Inactive → Active
func sceneWillResignActive(_ scene: UIScene) { }      // Active → Inactive (e.g. incoming call)
func sceneDidEnterBackground(_ scene: UIScene) { }    // save state, stop timers
func sceneWillEnterForeground(_ scene: UIScene) { }   // refresh stale data
```

**Inactive** is the transient state during a transition (a phone call arriving, the app switcher opening) — the UI is visible but not receiving events; **Background** means no UI is visible but code can still run briefly (or via a registered background task); **Suspended** means frozen in memory with no CPU time, and the OS can purge it under memory pressure without notice — which is why persisting critical state happens on the *background* transition, not suspend.

**Say it:** "AppDelegate owns process-launch, SceneDelegate owns per-window UI transitions since multi-window iPad split those apart — I persist state on `sceneDidEnterBackground`, not suspend, because suspended apps can be purged from memory without another callback."
**Red flag:** Relying on a suspend callback to save state. There isn't a dedicated one — the app can be silently killed straight from suspended; background entry is the last guaranteed hook.

### APNS, local vs remote notifications, and notification extensions
**They ask:** "How does Apple Push Notification service actually work end to end, and what do service/content extensions let you do with a payload before it's shown?"

The reason push exists instead of polling is battery and immediacy: a **remote** notification is generated by your server, sent to APNs over a persistent authenticated connection (token-based auth via a `.p8` key is the modern approach), and APNs pushes it to the device even when your app isn't running — the OS wakes just enough to display it or hand it to your extension. A **local** notification is scheduled entirely on-device (`UNUserNotificationCenter.add`) with no server or network involved — for reminders, timers, anything the app itself knows about in advance.

```swift
let content = UNMutableNotificationContent()
content.title = "Walk reminder"
let trigger = UNTimeIntervalNotificationTrigger(timeInterval: 3600, repeats: false)
UNUserNotificationCenter.current().add(
    UNNotificationRequest(identifier: "walk", content: content, trigger: trigger)
)
```

Two app extensions intercept a *remote* push before the user sees it. A **Notification Service Extension** runs briefly when a "mutable-content" push arrives, letting you decrypt a payload, download an image, or modify the alert text before display — needed because a raw push payload is capped around 4KB, so rich content is usually fetched, not embedded. A **Notification Content Extension** customizes the *UI* shown when the user long-presses/expands a delivered notification — a custom view instead of the system default.

A **silent push** (`content-available: 1`, no alert) wakes the app briefly in the background to fetch fresh data with no user-visible banner — subject to OS throttling if abused. **Badges** are just a count field in the payload; the device token that identifies where to deliver a push can regenerate (reinstall, restore, OS update), so servers must handle registering a *new* token, not assume it's stable forever.

**Say it:** "Remote pushes go through APNs from my server and can wake the app even when it's not running; local notifications are scheduled entirely on-device with no server involved. A Notification Service Extension mutates the payload before display — usually to fetch rich content the ~4KB payload cap can't hold — and device tokens aren't stable forever, so my backend re-registers on every launch."
**Red flag:** Assuming a device token never changes once obtained. It can regenerate on reinstall or restore — sending to a stale token silently fails, so re-register on every app launch.

### Security — App Transport Security, private data storage, and biometrics
**They ask:** "What is App Transport Security, when would you add an exception, and how do you responsibly store and access private user data?"

ATS is a default-on OS policy that **rejects insecure HTTP connections** and enforces minimum TLS versions/cipher strength for every request an app makes — it exists because a networked app is otherwise an easy target for a network-level attacker to downgrade or intercept traffic. Adding an `NSExceptionDomains`/`NSAllowsArbitraryLoads` entry to `Info.plist` should be a *last resort*, scoped to a specific domain you don't control (a legacy internal server), never a blanket opt-out — App Review will ask you to justify it.

```xml
<key>NSAppTransportSecurity</key>
<dict>
  <key>NSExceptionDomains</key>
  <dict>
    <key>legacy-internal.example.com</key>
    <dict><key>NSExceptionAllowsInsecureHTTPLoads</key><true/></dict>
  </dict>
</dict>
```

For private data at rest: the sensitivity determines the store — Keychain for credentials/tokens (encrypted, survives reinstall), file-level encryption (Data Protection classes like `.complete` or `.completeUntilFirstUserAuthentication`) for sensitive files, and explicit **user consent** before touching anything the App Store's privacy rules gate — location, contacts, photos, health data — each requiring an `Info.plist` usage-description string and a runtime permission prompt the user can deny.

Biometric auth (`LocalAuthentication`'s `LAContext`, Touch ID/Face ID) gates access to an already-stored secret rather than being the secret itself — it's a local *presence check*, and the actual protected data still lives behind Keychain's access control, ideally with `kSecAccessControlBiometryCurrentSet` so a newly-enrolled fingerprint/face invalidates access.

**Say it:** "ATS blocks plaintext HTTP by default and any exception should be scoped to one domain I don't control, never a blanket opt-out App Review will flag. For private data, sensitivity picks the store — Keychain for secrets, Data Protection classes for sensitive files — and biometrics gate access to that stored secret rather than being the secret, with `kSecAccessControlBiometryCurrentSet` so a re-enrolled face invalidates old access."
**Red flag:** Setting `NSAllowsArbitraryLoads` to unblock a networking bug during development and shipping it. That disables TLS enforcement app-wide — App Review rejects it, and it's a real security hole even before review catches it.

### Deep linking — URL schemes, Universal Links, and third-party services
**They ask:** "Custom URL schemes versus Universal Links — what are the actual trade-offs, and why would you still add something like Branch.io on top?"

A custom **URL scheme** (`pawwalk://profile/42`) is registered by the app in `Info.plist` and works even without internet, but it's not exclusive — any app can register the same scheme, so a malicious app can potentially hijack it (scheme squatting), and if the target app isn't installed the link just fails silently with no fallback. **Universal Links** (`https://pawwalk.app/profile/42`) are ordinary HTTPS URLs verified via a signed `apple-app-site-association` file hosted on your domain — Apple checks that file at install/link time, so only the domain owner can claim the link, and if the app isn't installed the link gracefully falls through to the actual web page instead of failing.

```
# apple-app-site-association (served over HTTPS, no extension)
{ "applinks": { "details": [{ "appID": "TEAMID.com.pawwalk.app", "paths": ["/profile/*"] }] } }
```

The trade-off in one line: custom schemes are simpler to set up and work offline but are spoofable and have no graceful fallback; Universal Links are secure and gracefully degrade to web, but require domain control and a hosted, correctly-signed association file — which is exactly the setup step teams skip and then wonder why links open Safari instead of the app.

Third-party deep-linking services (Branch.io and similar) exist to solve what neither native mechanism does alone: **attribution and deferred deep linking** — if the app *isn't* installed, they route through the App Store, capture install context, and route the user to the right in-app screen on first open, plus give unified analytics/short-link generation across iOS and Android. Native links can't do deferred linking on their own because there's no mechanism to reference "what link led here" after an App Store detour.

**Say it:** "Universal Links are domain-verified HTTPS links that gracefully fall back to the web and can't be spoofed; custom schemes are simpler and work offline but are squattable and fail silently without a fallback. I'd add Branch.io specifically for deferred deep linking — routing a not-yet-installed user to the right screen after their first open — which neither native mechanism does on its own."
**Red flag:** Recommending only a custom URL scheme for a marketing/sharing link. Any other app can register the same scheme, and there's no fallback if your app isn't installed — Universal Links (or a service layered on top) is the correct default for links leaving the app.
