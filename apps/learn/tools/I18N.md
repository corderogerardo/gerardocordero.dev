# Translating the courses (en → es)

The `/es` UI chrome is bilingual, but `public/data/es/*.json` are currently
copies of the English courses. This pipeline translates them incrementally,
module by module, without ever touching code.

## Scope (measured)

| course  | modules | lessons | steps | prose words |
|---------|--------:|--------:|------:|------------:|
| ios     | 21      | 84      | 585   | ~53,500     |
| android | 21      | 87      | 509   | ~47,100     |
| ruby    | 24      | 135     | 856   | ~72,100     |
| python  | 28      | 126     | 664   | ~54,200     |
| go      | 14      | 41      | 278   | ~23,300     |
| **all** | **108** | **473** | **2,892** | **~250,000** |

Code fields (`starter`, `solution`, `checks`, `mustNot`, `source`) are never
extracted — the merge copies them verbatim from English. Fenced code inside
`md` is only ~2% of md words; the translator must leave it untranslated.

## Pipeline

```bash
node tools/i18n-extract.mjs go            # list modules + word counts
node tools/i18n-extract.mjs go 0          # -> tools/i18n-work/go/00-<id>.json
# translate every `text` in that file (keep `path`, code fences, URLs as-is)
node tools/i18n-merge.mjs go              # -> public/data/es/go.json
```

Merge always starts from the English course and applies whatever work files
exist, so partially-translated courses are valid: untranslated modules simply
render in English. Re-running merge is idempotent.

## Suggested execution

One module ≈ 1–4k words — a good unit for one cheap-model (sonnet/haiku)
agent pass. Order by impact: **go** (smallest, proves the loop) → ios →
android → python → ruby. After each merge run `pnpm --filter
@gerardocordero/learn typecheck` and click through `/es/learn/<course>`.
